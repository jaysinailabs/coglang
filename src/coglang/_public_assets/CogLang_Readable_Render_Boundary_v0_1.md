# CogLang Readable Render Boundary v0.1

**Status**: public governance note
**Scope**: boundary between canonical text, human-readable rendering, and host transport envelopes
**Audience**: maintainers, contributors, tooling authors, UI authors, and host implementers
**Non-goal**: implementing a readable renderer or changing canonical text, parser, validator, or transport semantics

---

## 0. How To Read This Document

`canonical text` is the current stability anchor for CogLang expression values.
It is the parseable, replayable, diffable, and testable text form produced by
`canonicalize`.

`readable render` is a future human-facing display surface. It may make CogLang
expressions easier to scan in user interfaces, traces, reviews, or editor views,
but it is not currently a separate public serialization contract.

This document records the boundary that future readable rendering work must
respect. It does not:

- add a new syntax
- require the parser to accept a new text form
- add a CLI command or option
- change `canonicalize`
- change validation or conformance behavior
- change host-runtime envelopes or HRC v0.2

The practical rule is simple: readable rendering may improve human inspection,
but canonical text and structured host data remain the machine-facing anchors
until a future specification and conformance update says otherwise.

---

## 1. Terms

| Term | Meaning | Current stability |
| --- | --- | --- |
| CogLang value | Parsed expression, variable, literal, list, dict, or related in-memory value used by the implementation. | Stable only through documented parser, canonicalization, validation, execution, and host surfaces. |
| Canonical text | Stable M-expression-style text generated by `canonicalize`. | Current language text anchor. |
| Readable render | A human-facing display of a CogLang value that may use spacing, indentation, abbreviations, highlighting, grouping, or other presentation choices. | Not yet a public serialization contract. |
| Transport envelope | Structured host/runtime message or record, such as write-bundle submission and response data. | Governed by the relevant host-runtime contract or companion classification. |

Readable rendering is a view over language or host data. It is not the owner of
that data.

---

## 2. Boundary Rules

Future readable rendering work should follow these rules.

### 2.1 Canonical Text Remains The Replay Anchor

If a value needs to be parsed again, hashed, diffed, cached, signed, replayed,
embedded in a conformance fixture, or exchanged between machines as CogLang
text, use canonical text.

Readable rendering may be shown next to canonical text, but it must not silently
replace canonical text in machine-facing paths.

### 2.2 Readable Rendering Is Display-Only Until Promoted

A readable render may be useful for:

- UI panels
- logs intended for humans
- review artifacts
- trace summaries
- editor previews
- tutorial or explanation views

Those uses do not imply that the render is parseable, stable across versions, or
safe as the only stored representation. If a tool exports a readable render, the
export should be labeled as display output unless it is also valid canonical
text.

### 2.3 Transport Envelopes Stay Structured

Host/runtime exchange should use structured fields and documented envelope
objects. A transport envelope must not require consumers to parse a readable
render in order to recover machine data.

When a host-visible artifact needs a textual expression, the canonical text or
structured value should remain available. Readable text can be an auxiliary
human-facing field only.

### 2.4 UI Consumers Should Preserve A Canonical Escape Hatch

User interfaces may default to readable rendering, but they should preserve a
way to inspect or copy canonical text for debugging, bug reports, tests, and
cross-tool handoff.

If a UI lets a user edit readable text, the acceptance path must be explicit:
either the text is canonical text, or a future parser/formatter specification
defines the accepted readable form.

### 2.5 Formatting Options Are Not Semantics

Indentation, color, grouping, line breaks, elision, truncation, or label choices
must not change the meaning of a CogLang value.

Any readable renderer that hides data for display should clearly retain access
to the full canonical or structured value.

## 3. Current Implementation Posture

The current public implementation provides canonicalization, not a separate
readable renderer.

Current stable expectations:

- `canonicalize` is the text rendering function with round-trip tests.
- `parse(canonicalize(expr)) == expr` is the compatibility expectation for
  supported values.
- CLI commands that print expression results use canonical text unless a
  structured JSON format is requested.
- host demos and write-bundle paths use structured message objects and canonical
  expression text where text is needed.

This posture keeps the language core small while leaving room for future display
work.

---

## 4. Future Promotion Artifact Set

A future PR that promotes readable rendering from boundary note to public
subsystem should include at least:

1. A readable rendering specification that defines scope, options, and stability.
2. Golden examples comparing parsed value, canonical text, and readable render.
3. A policy for whether readable output is parseable, non-parseable display, or
   both under different modes.
4. Tests proving that canonical text and existing parser behavior are unchanged.
5. CLI or API documentation if a public command, flag, or function is added.
6. UI/tooling guidance for preserving canonical text and structured data.
7. README, `llms.txt`, `llms-full.txt`, manifest, package data, public extract,
   and release-check updates if the public surface changes.

Promotion should be narrow. A readable renderer can become a useful subsystem
without changing the language grammar.

---

## 5. Review Checklist

Readable rendering proposals should answer these questions before merge:

- Does the change alter `canonicalize` output?
- Does it alter parser acceptance?
- Does it introduce a second machine-facing serialization format?
- Does it hide canonical text from users who need debugging or handoff?
- Does it require host transport consumers to parse display text?
- Does it change HRC or write-bundle envelope semantics?
- Are golden examples and regression tests included for every new public mode?
- Is the output clearly labeled when it is display-only?

If the answer to any boundary question is unclear, keep the work as
documentation, examples, or private UI code until the semantic and conformance
story is explicit.

---

## 6. Non-Goals

This boundary note does not:

- define pretty-print syntax
- define an editor formatter
- define a second parser mode
- define a transport serialization
- make readable render output stable release data
- change `CogLang_Specification_v1_1_0_Draft.md`
- change `CogLang_HRC_v0_2_Final_Freeze_2026_04_28.md`
- change the public conformance suite

The goal is to prevent future display work from blurring the existing split
between canonical language text, human-facing presentation, and structured
host/runtime exchange.
