# CogLang Vision Proposal v0.1

**Status**: public vision proposal
**Scope**: long-term capability direction, not a release contract
**Audience**: users, contributors, host implementers, and AI-system builders evaluating where CogLang may go next

---

## 0. How To Read This Document

This document is a proposal, not a promise.

The stable language line remains `v1.1.0`. The current package line is `1.1.x`.
The Host Runtime Contract v0.2 is frozen only for the narrow typed
write-envelope surface documented in
`CogLang_HRC_v0_2_Final_Freeze_2026_04_28.md`.

This proposal describes the direction CogLang should grow toward if it continues
to serve AI systems that generate graph operations.

It should not be read as:

- a dated roadmap
- a v1.2 release checklist
- a guarantee that every listed capability will ship
- a claim that CogLang replaces agent frameworks, graph databases, tool
  protocols, or observability stacks

---

## 1. Long-Term Identity

CogLang should grow into an auditable semantic-action intermediate language for
AI-generated graph operations.

The short form is:

> CogLang lets AI-generated graph actions become inspectable, replayable, and
> rejectable before they affect a host graph.

The long-term goal is not to become a general-purpose programming language. It
is to become a stable intermediate representation for graph-intent operations
that can be generated by a model, checked by a host, reviewed by a human or
policy engine, and replayed later as evidence.

---

## 2. Why This Direction Matters

AI systems are moving from text generation toward action:

- tool calls
- long-term memory writes
- knowledge graph updates
- workflow handoffs
- agent-to-agent delegation
- host-side policy decisions
- compliance-oriented audit trails

Agent frameworks, tool protocols, and observability systems help with these
layers, but they do not by themselves define a compact symbolic language for
the semantic graph action an AI system intended to perform.

CogLang should occupy that gap.

It should complement:

- agent frameworks, by representing graph-level intent inside a workflow
- tool protocols, by giving tools a preflightable semantic payload
- observability stacks, by giving traces a canonical semantic sidecar
- graph databases, by expressing host-neutral graph intent before native
  database execution

## 3. Core Design Commitments

The following commitments should remain stable unless a future major language
version intentionally changes them.

### 3.1 Canonical Text Is The Stability Anchor

Canonical M-expression text must remain the stable representation used for
storage, hashing, diffing, replay, examples, and conformance.

Readable rendering may improve, but it must not replace canonical text as the
authoritative representation.

### 3.2 Errors Remain Values

Failures should remain explicit values, not hidden host exceptions or silent
null-like control flow.

This is especially important for AI-generated input, where partial failure is
normal and should be inspectable.

### 3.3 Core Stays Small And Graph-First

Core should remain centered on graph queries, graph updates, rule objects,
observable execution, and host boundaries.

Capabilities that mainly serve a specific host, workflow, or application domain
should usually enter through profiles, examples, adapters, or roadmap notes
before they are considered for Core.

### 3.4 Profiles And Capabilities Carry Extension Pressure

CogLang should not solve every domain-specific requirement by expanding Core.

Profiles and capability declarations should be the normal path for host-specific
or application-specific behavior.

### 3.5 Expression Values Stay Inspectable

Expression values should remain suitable for canonical text, readable rendering,
hashing, diffing, and replay.

Large opaque payloads such as raw binary blobs, tensors, media bytes, or model
artifacts should normally be represented by host-owned references plus metadata,
not embedded directly as expression payloads.

## 4. v1.2 Candidate Themes

The following themes are plausible candidates for a future `v1.2` planning
cycle. They are not commitments.

### 4.1 Effect And Capability Preflight

CogLang should be able to describe what an expression may require before a host
executes it.

Candidate preflight categories:

- graph read
- graph write
- graph delete
- traversal
- unification
- host capability requirement
- external tool or adapter requirement
- human review requirement
- possible `ErrorReport`

The goal is not to prove every runtime behavior statically. The goal is to give
hosts enough structured evidence to reject unsafe or unsupported operations
before execution.

### 4.2 Review Bundle

CogLang should provide a standard review artifact for AI-generated graph
actions.

A review bundle could contain:

- canonical expression
- readable explanation
- planned graph diff
- required capabilities
- risk flags
- estimated graph-compute budget
- correlation identifiers
- expected success envelope or possible `ErrorReport`

This would make "inspect before execute" concrete for humans, policy engines,
and host integrations.

### 4.3 Agent Memory And Knowledge-Graph Write Preflight

The first high-value demo should show an AI system proposing a long-term memory
or knowledge graph update:

1. The model proposes a CogLang expression.
2. CogLang parses, canonicalizes, and validates it.
3. The host estimates effects, required capabilities, and graph-compute budget.
4. A human or policy engine approves or rejects the write.
5. The host submits the typed write envelope.
6. The result is traceable and replayable.

This demo should stay small and runnable.

### 4.4 Observability Sidecar

CogLang should be easy to attach to existing traces without replacing
observability stacks.

Potential sidecar fields:

- `coglang.canonical`
- `coglang.hash`
- `coglang.language_release`
- `coglang.effect_summary`
- `coglang.correlation_id`
- `coglang.submission_id`
- `coglang.payload_kind`

This should remain an integration pattern, not a requirement that all hosts use
one tracing vendor or telemetry protocol.

### 4.5 Tool Protocol Preflight

CogLang should fit naturally in tool protocols and agent frameworks as a
semantic payload layer.

A minimal tool-wrapper example could:

1. receive a CogLang expression
2. parse and validate it
3. run effect and capability preflight
4. apply host policy
5. execute only after approval

This should not make CogLang a replacement for tool protocols. The goal is to
make tool calls safer and more inspectable when the target action is graph-like.

## 5. Graph Compute Governance

Graph computation has unique failure modes. A small-looking graph expression can
become computationally unreachable if traversal, unification, recursion, path
expansion, or result cardinality is not bounded.

CogLang should treat graph-compute reachability as a first-class design concern.

### 5.1 Why Graph Compute Needs Its Own Boundary

Graph operations can become expensive because of:

- high-degree nodes
- recursive traversal
- path explosion
- unbounded joins or unification
- large result sets
- repeated subgraph expansion
- hidden host-side graph indexes or missing indexes

For AI-generated input, this risk is larger because a model can produce a valid
expression without understanding the host graph's size, density, or indexing
constraints.

### 5.2 Proposed Budget Model

Future CogLang host integrations should be able to attach or enforce a
graph-compute budget.

Candidate budget fields:

- max traversal depth
- max visited nodes
- max visited edges
- max result count
- max path count
- max recursion depth
- max unification branches
- max intermediate bindings
- max execution time
- max memory estimate
- host-specific cost class

The budget should be host-enforced. The language can describe budget intent and
standard error behavior, but the host decides what is safe for its graph.

### 5.3 Budget Errors As Values

Budget failures should return structured values, not host crashes or silent
truncation.

Candidate error categories:

- `BudgetExceeded`
- `TraversalLimitExceeded`
- `ResultLimitExceeded`
- `PathExplosion`
- `Timeout`
- `HostCostUnsupported`

The exact spelling is not frozen by this proposal.

### 5.4 Preflight And Explainability

Hosts should be able to provide a conservative preflight result before
execution:

- known safe
- likely safe under a named budget
- requires approval
- rejected because the cost is unbounded
- rejected because the host cannot estimate the cost

This preflight should be visible in review bundles and traces.

### 5.5 Non-Goal

CogLang should not become a distributed graph compute engine.

It should provide:

- a bounded expression layer
- cost and effect vocabulary
- host-visible budget contracts
- standard failure envelopes

It should not own:

- graph storage
- distributed query planning
- index selection
- GPU scheduling
- cluster-level graph execution

## 6. AI Learning And Generation Maturity

CogLang is designed to be generated by AI systems, but that does not mean AI
systems automatically learn it well.

The project should treat "AI can learn and generate CogLang reliably" as an
engineering surface with tests and maturity levels.

### 6.1 Learning Materials

AI-facing learning materials should include:

- README summary
- `llms.txt`
- `llms-full.txt`
- Quickstart
- operator catalog
- conformance examples
- invalid examples and repair examples
- prompt-to-expression tasks
- expression-to-explanation tasks
- error-guided repair tasks

The goal is not to create private training claims. The goal is to make the
public surface learnable and testable.

### 6.2 Generation Maturity Levels

The project can evaluate AI generation maturity in levels:

| Level | Name | Meaning |
| --- | --- | --- |
| L0 | Recognition | The model can identify what CogLang is for. |
| L1 | Syntax | The model can produce parseable M-expressions. |
| L2 | Canonicalization | The model can produce or repair canonical-compatible expressions. |
| L3 | Validation | The model can avoid common arity, operator, and value-shape errors. |
| L4 | Execution | The model can produce expressions that execute correctly in the reference runtime. |
| L5 | Host Awareness | The model can include capability, effect, and host-boundary considerations. |
| L6 | Review Awareness | The model can produce expressions plus explanations suitable for human or policy review. |

### 6.3 Public Evaluation Metrics

Useful public metrics include:

- parse pass rate
- validation pass rate
- canonical roundtrip pass rate
- execution pass rate
- conformance scenario pass rate
- repair success after one error message
- hallucinated operator rate
- unsafe write-intent rate
- missing capability declaration rate
- budget-preflight rejection rate

These metrics should be public, reproducible, and modest. The project should not
publish private accuracy claims as public evidence.

### 6.4 AI Learning Roadmap

Future work may include:

- a small public prompt-to-CogLang eval
- a repair benchmark using structured error outputs
- a "CogLang for LLMs" prompt card
- generation-focused examples in `llms-full.txt`
- conformance-derived training pairs
- host-aware generation examples

This work should help both humans and models understand the language boundary.

## 7. Long-Term Capability Map

CogLang should grow in layers.

### Layer 1: Stable Language Core

- canonical text
- parser / validator / executor consistency
- core graph operators
- errors as values
- conformance examples

### Layer 2: Effect And Budget Preflight

- effect summaries
- capability requirements
- graph-compute budgets
- standard budget errors
- host-side rejection reasons

### Layer 3: Review And Evidence

- review bundles
- planned graph diffs
- trace and replay evidence
- host submission records
- error reports

### Layer 4: Host And Tool Integrations

- reference host
- non-reference host examples
- tool-wrapper preflight
- observability sidecar
- agent memory / knowledge graph write demo

### Layer 5: AI Learning Surface

- prompt-to-expression examples
- repair examples
- generation evals
- machine-readable summaries
- conformance-derived examples

---

## 8. What Success Looks Like

CogLang is successful if an AI system can propose a graph action and the host
can answer these questions before execution:

1. What exact expression did the AI produce?
2. What graph action does it intend?
3. Is the expression valid?
4. What effects and capabilities does it require?
5. Is the graph-compute cost bounded?
6. What will likely change?
7. Can a human or policy engine approve or reject it?
8. What happened after execution?
9. Can the result be replayed or audited later?

If CogLang can answer those questions with a small, stable language and a
repeatable host contract, it has a durable role in the AI-agent ecosystem.

---

## 9. Near-Term Recommendation

The next focused work should be:

1. design a minimal effect and graph-budget preflight vocabulary
2. implement a review-bundle prototype for the current host demos
3. create an agent memory / knowledge-graph write preflight demo
4. create a small public AI-generation eval

These items are more important than adding broad new syntax. They move CogLang
toward its strongest long-term role: a small, inspectable, AI-generated semantic
action IR for graph operations.
